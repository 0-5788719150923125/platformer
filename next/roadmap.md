# Platformer Roadmap

Tracked work items for the Platformer framework. These range from near-term wins to longer-term architectural goals. Items are roughly ordered by priority, not dependency.

> **Note on Licensing**: There is ongoing discussion about adding an open source license to this project. The HashiCorp Business Source License (BSL) is being considered as a potential option, as it would allow source-available distribution while providing some protections. This decision is still under evaluation and requires team alignment before proceeding.

---

- [x] **Build AMIs automatically (Packer)**
  Automate golden image creation so base AMIs are versioned, tested, and reproducible rather than hand-built snapshots that drift over time. Initially implemented with AWS ImageBuilder, then migrated to Packer for multi-cloud capability and simpler local development. Packer builds are triggered by the compute module when a class declares `build: true`, producing golden AMIs that arrive pre-hardened instead of patched after launch. This closes the loop between patch management and compute provisioning.

- [ ] **Cross-region EKS clusters**
  Deploy EKS clusters across multiple regions from a single state fragment as proof of multi-region orchestration. The networking module already supports multi-VPC and the compute module routes by type, so the plumbing is largely in place. This validates the framework's claim that regional expansion is a configuration change, not an architecture change.

- [x] **Portal scorecards for patch management**
  Scorecards evaluate patch management configuration maturity (Basic → Bronze → Silver → Gold) on EC2 entities. A compliance reporter Lambda polls SSM patch states every 15 minutes and pushes runtime compliance data (COMPLIANT/NON_COMPLIANT, installed/missing counts, last scan time) to Port via webhook. A dashboard pie chart shows patch readiness distribution. The generic lambda-requests interface on the configuration-management module supports future scheduled Lambdas via dependency inversion.

- [ ] **HMAC signature verification for Port webhook**
  The compliance reporter webhook currently accepts unauthenticated POSTs from the Lambda function. Add HMAC-SHA-256 signing in the Lambda (shared secret via SSM Parameter Store or Secrets Manager) and configure the `security` block on the `port_webhook` resource to verify signatures. This prevents unauthorized entities from pushing compliance data into the portal.

- [x] **Add entitlements to tenants module**
  Tenants now declare entitlements in `matrix.tenants` state fragments using dot-notation: bare service names (e.g., `compute`) grant access to all classes, scoped names (e.g., `compute.windows-server`) grant access to specific classes. The tenants module resolves these into `tenants_by_service` and `tenants_by_class` outputs that consumer modules use directly. Compute, archshare, and archpacs derive their tenant lists from entitlements rather than from hardcoded lists in compute config.

- [x] **Move tenants from services.tenants to matrix.tenants**
  Elevated tenants to a top-level organizational concept by moving tenant definitions from `services.tenants` to `matrix.tenants`. Regions and tenants are now peers under the `matrix` key, recognizing them as the two primary targeting dimensions for deployment isolation. The config module exposes a `matrix_configs` output alongside `service_configs`, and the tenants module reads from `matrix.tenants`. All state fragments (production and test) were migrated. Module interfaces and entitlement syntax are unchanged.

- [ ] **Integrate tenants module with ServiceNow**
  Pull tenant records and entitlements from ServiceNow rather than maintaining them as static configuration. This connects the framework to the organizational source of truth for tenant lifecycle, ensuring that onboarding, offboarding, and entitlement changes propagate automatically.

- [X] **Expand integration test scenarios with business use cases**
  The integration testing framework is operational — 3 scenarios, 23 passing tests validating fragment combinations, dependency inversion, and deterministic CIDR allocation. Next step is adding scenarios that model real business deployments: multi-tenant Linux fleets with archival storage, privacy-constrained Windows deployments without S3 logging, and multi-region Kubernetes with disaster recovery. Tests should read as user stories that map directly to tenant requests.

- [x] **Event logging with the LGTM stack**
  Establish a centralized logging and observability pipeline using Grafana's LGTM stack (Loki, Grafana, Tempo, Mimir) with Grafana Alloy as the collection agent. The company is converging on Grafana as the observability platform, so aligning here avoids a future migration. Alloy replaces a grab-bag of per-service log shippers with a single, declarative collector that can forward logs, metrics, and traces. Initial scope would be shipping CloudWatch and VPC flow logs into Loki, with Mimir for infrastructure metrics and Tempo for distributed tracing as follow-on layers. State fragments should own the Alloy configuration so each deployment composes its own telemetry pipeline the same way it composes compute or networking.

- [ ] **Minimize cross-account dependencies**
  Cross-account segregation of duties is often cited as a best practice, but the operational complexity frequently outweighs the security benefit. Each cross-account boundary adds IAM roles, trust policies, and assume-role chains that make the system harder to reason about, troubleshoot, and maintain. The framework should default to single-account deployments and only introduce cross-account patterns when there's a clear, measurable threat model that justifies the complexity. Modern AWS primitives (SCPs, permission boundaries, resource-based policies) provide strong isolation within a single account. Delivery velocity and maintainability are security properties — simpler systems are easier to audit and less prone to misconfiguration.

- [ ] **Microsoft Teams notifications via Power Automate**
  Send operational notifications (patch completions, deployment events, action results) to Microsoft Teams channels. Rather than relying on manually-created, hardcoded incoming webhook URLs — which break when channels move and can't be managed as code — the framework should provision the webhook itself through the Power Automate Management API. A Power Automate flow acts as the bridge: Terraform creates the flow (HTTP trigger → Post to Teams), captures the trigger URL as an output, and passes it to modules that need to send notifications. This keeps the entire lifecycle in code — no portal clicks, no shared secrets pasted into tfvars. The configuration-management module is the first consumer (patch window start/complete, baseline drift), with portal actions and compute provisioning as follow-on senders. State fragments would declare a `notifications` block specifying channel targets and event filters, and the module would compose one flow per channel.

- [ ] **Evaluate open-sourcing the Platformer framework**
  Platformer contains zero proprietary code — no tenant names, no internal domains, no account IDs. The framework is purely abstract patterns: state fragments, dependency inversion, module composition. Tenant-specific configuration (top.yaml, real state fragments, tenant registry) stays private. Organizations face the same Terraform scalability problems we solved, and publishing would invite external scrutiny that pressure-tests the design. Requires team discussion before proceeding.

- [ ] **Evaluate alternative tooling**
  Assess whether ArgoCD, Argo Workflows, or Terragrunt align with Platformer's design principles. ArgoCD could manage application deployments on EKS clusters (complementary to Platformer's infrastructure layer), following the Unix philosophy of specialized tools that compose well. Argo Workflows could replace scheduled Lambdas with Kubernetes CronWorkflows, consolidating compute, though Terraform already handles dependency orchestration and GitHub Actions manages CI/CD. Terragrunt solves problems (DRY config, remote state, cross-module dependencies) that Platformer addresses through state fragments and dependency inversion — adding it risks layering a second abstraction system on existing patterns. Only adopt tools when pain of not having them exceeds the cost of integration and maintenance.

- [x] **EKS Pod Identity support**
  EKS Pod Identity replaces IRSA for granting pods access to AWS services. Simpler than IRSA: no OIDC provider, no two-apply dependency, just a trust policy on `pods.eks.amazonaws.com` and an `aws_eks_pod_identity_association` resource. The compute module provisions the `eks-pod-identity-agent` addon and handles associations generically — upstream modules emit `pod_identity_requests` (name, cluster class, namespace, service account, IAM policy) and compute creates the roles and bindings. Currently used by the EBS CSI driver addon and Loki S3 access in the observability module.

- [X] **Standardize and clean up Terraform outputs**
  The root and module outputs have grown organically and are cluttered with implementation details that belong in state, not in terminal output. Full resource metadata (AMI IDs, subnet IDs, IAM role ARNs, security group maps) is duplicated across outputs when it's already queryable via `terraform state show` or `terraform output -json`. The outputs should be restructured around three tiers: (1) **essential** — deployment ID, region, enabled services, and access URLs that operators need immediately after apply; (2) **operational** — command registries and service health checks for day-two tasks; (3) **programmatic** — structured data for portal and downstream consumers, hidden from terminal via `sensitive = true` or consumed only by other modules. Module outputs that exist purely for dependency inversion (instance keys, application requests, compute class requests) should not propagate to root. Per-instance metadata like `ingress_ports`, `all_ports`, and `instance_type` in `service_urls` adds noise without value — the portal can fetch these from state if needed. `data "external"` outputs (Grafana URL, Loki gateway URL) should be replaced with proper Terraform-managed resources or moved behind a dedicated lookup module to avoid shell-exec dependencies in the output layer.

- [X] **CrowdStrike Falcon agent deployment**
  Automated CrowdStrike Falcon agent installation via Ansible playbook (`applications/ansible/crowdstrike/`). Uses the official CrowdStrike installer script with OAuth2 credentials from Secrets Manager. A generic cross-account secret replication mechanism (`secrets.tf`) reads credentials from `example-infrastructure-prod` and writes local copies to the deployment account — instances read from local Secrets Manager at runtime, no cross-account IAM needed. The playbook skips already-healthy installations and verifies agent registration post-install. Enablement is via state fragment: add `crowdstrike` to the applications list on any compute class. Windows support and sensor proxy configuration for private subnets are follow-on items.

- [X] **Parallelize Ansible controller execution for cluster deployments**
  The orchestrator (`orchestrator.py`) runs manifest entries sequentially, which is fine for independent playbooks but slow for `mode: 1-master` clusters where N instances each get their own entry. Either add `concurrent.futures` parallelism in the orchestrator for entries targeting different instances, or restructure cluster entries into a single playbook run with per-host variables so Ansible's native fork-based parallelism handles it. The latter is more Ansible-idiomatic but requires changes to how compute emits cluster application requests.

- [X] **Expand taint SSAs to EKS/ECS clusters and pods**
  The portal's self-service taint action currently supports EC2 instances only. Extend the pattern to support tainting EKS node groups, ECS services/tasks, and individual Kubernetes pods. For EKS, this could taint `aws_eks_node_group` resources to force node rotation; for ECS, taint `aws_ecs_service` to force task redeployment; for pods, issue a `kubectl delete pod` via the action handler rather than Terraform state manipulation. The command registry and action handler already support pluggable action types (`state_taint`, `cli_exec`), so the plumbing is in place — each new target type needs a command definition in the relevant module's outputs and a corresponding handler in `app.py`.

- [X] **Rename "customers" to "tenants"**
  The `customers` module and associated terminology (`services.customers`, `customers_by_service`, `customers_by_class`) imply a commercial relationship that doesn't match actual usage. Current "customers" include `test` (testing), `platform` (internal department), and `cimar` (a partner/vendor) — none of which are customers in any meaningful sense. Renaming to `tenants` better reflects the module's actual purpose: isolating workloads by organizational boundary regardless of the relationship type. This is a rename across the module, state fragments, `top.yaml`, outputs, and portal blueprints. No behavioral change.

- [X] **Portal events widget (notification bus)**
  Add a generic events blueprint and dashboard widget that aggregates notifications from multiple sources, similar to how the compute widget lists different kinds of compute (EC2, EKS, ECS) under one interface. Each event has a source type, timestamp, status, and message — the blueprint routes by source the same way compute routes by type. First consumer is CodeBuild build events (start, succeed, fail, phase transitions), pushed to Port via webhook from EventBridge. Future sources include scheduled Lambda completions, Ansible playbook results, patch compliance state changes, and portal action outcomes. The widget displays a time-sorted feed filtered by namespace, giving operators a single pane for "what happened recently" without switching between CloudWatch, CodeBuild console, and portal action logs. State fragments declare which event sources to enable per deployment.

- [ ] **Evaluate public Terraform modules for networking**
  Assess whether the community terraform-aws-modules/vpc module and related networking modules align with Platformer's deterministic CIDR allocation and multi-region patterns. The public modules are battle-tested and maintained by the community, potentially reducing custom code maintenance burden. However, they may not support the hashing-based CIDR generation and deep-merge state fragment patterns that enable Platformer's configuration composability. Compare feature parity, flexibility for our use cases, and whether adopting them simplifies or complicates the networking module's interface.

- [ ] **Research module - scientific LaTeX reporting and formal verification**
  Add a `research` module that generates scientific LaTeX documents from templates, driven by data the framework already produces. A single base template covers the initial report types: drift detection, change propagation speed, documentation currency, control verification, authorship dependencies, audit documentation, operational continuity, and domain specialist review. The template is data-driven - Terraform outputs and state metadata feed into it, rather than hand-authoring reports around fixed conclusions. Longer term, model a subset of these properties in Lean so that invariants (e.g., "every apply reduces drift," "documentation age never exceeds N days post-change") can be machine-verified as formal proofs. These proofs would run as atomic Terraform operations, each advancing the system by one state transition and proving the transition preserves declared invariants. This connects infrastructure management to reproducible, publishable science rather than ad-hoc operational reporting.

- [ ] **Sequential time reversal via inverse state operations**
  Because `top.yaml` is additive and every operation has a perfect inverse through state manipulation, the framework can rewind time by playing the same lifecycle sequence backwards. Each forward state transition (create, modify, scale, reconfigure) has a hand-crafted inverse that cleanly reverts it, so rolling back to any prior point in the timeline is a sequential subtraction through the same `top.yaml` file in reverse order. The tradeoff is wall-clock time - you're replaying history backwards, one state at a time, and the cost scales linearly with how far back you need to go. But the guarantee is exact: every intermediate state is visited and verified on the way back, not skipped. This is distinct from restoring a state snapshot, which jumps to a point but can't prove the path between here and there was clean. The inverse-replay approach treats rollback as a first-class operation with the same rigor as forward deployment.

- [ ] **Rename "applications" to "automations"**
  The `applications` module and associated terminology (`application_requests`, `standalone_applications`, `application-scripts`) implies software installation, but the module also orchestrates runbooks (password resets, compliance remediations), Ansible playbooks that configure rather than install, and scheduled operational tasks that have nothing to do with application delivery. Renaming to `automations` better reflects the actual scope: any repeatable action that runs on or against infrastructure, whether that's installing software, rotating credentials, running a playbook, or executing a remediation script. This is a rename across the module directory, state fragments, variable names, output keys, S3 bucket purpose, and portal blueprints. Behavioral change: none. The `type` field values (`ssm`, `ansible`, `helm`, `user-data`) and targeting modes (`wildcard`, `tags`, `compute`) are unchanged.

- [ ] **Billing management dashboard in portal**
  Integrate a billing dashboard into the portal module to track accumulating costs of personally-deployed infrastructure. Query AWS Cost Explorer API to display current month-to-date spend, per-service breakdowns, and historical trends. Cache billing data to minimize API costs and provide budget alerts when spend exceeds thresholds.

- [ ] **Service showcase carousel on portal landing page**
  Build a CSS-animated iframe carousel in the portal's markdown dashboard widget that cycles through live service URLs, giving stakeholders a self-running visual tour of the deployment. Service URLs already flow into the portal via `service_urls` - the carousel would render the subset flagged with `showcase: true` in their state fragment metadata. Each slide is an `<iframe>` pointing at the real landing page (Grafana, application UIs, documentation sites), cross-fading on a timed loop using pure CSS keyframes - no JavaScript, no external dependencies. The `showcase` flag gives state fragment authors control over which services appear in the presentation: flip it on before a demo, flip it off after. The portal module builds the carousel HTML dynamically from the filtered `service_urls` map, so adding or removing a service from the rotation is a one-line state fragment change. This turns the portal home page into a living product demo that stays current with every apply, rather than a static screenshot deck that drifts the moment infrastructure changes.

- [ ] **Audit module - org-wide access control reporting**
  Add an `audit` module that collects access control declarations from every module via dependency inversion and produces a complete, org-wide JSON report as a static build artifact at apply time. Each module emits `access_requests` describing the permissions it provisions - IAM roles, policies, security group rules, S3 bucket policies, KMS grants, EKS RBAC bindings, SSM access, cross-account trust relationships - using a standardized schema. The audit module aggregates these into a single JSON document written to disk, giving every deployment a machine-readable snapshot of its entire permission surface. The report is deterministic and diffable, so changes between applies are visible in version control. The archivist module archives the report as a build artifact, and the portal module can render it as a searchable catalog of "who can access what." Longer term, the module could enforce policy-as-code rules (e.g., "no wildcard actions on production resources," "every cross-account role has an expiry condition") and fail the plan when violations are detected, turning the audit report from passive documentation into an active gate.
